Volatile的另一种理解

一段代码

public class VolatileTest &#123;

    // volatile static boolean flag;
    static boolean flag;

    public static void main(String[] args) &#123; throws InterruptedException &#123;
        new Thread ( () -> &#123;
            System.out.println ( "Thread start" );
            while (!flag) &#123;
                // todo something
            &#125;
            System.out.println ( "Thread end" );
        &#125; ).start ();
        Thread.sleep ( 1000 );
        flag = true;
        System.out.println ( "Main Thread end" );
    &#125;
&#125;

  • 写好后，单纯运行时发现子线程是无法停止的

  • 但是用volatile关键字修饰后，可以停止，得到结论是利用了volatile保证了可见性，但是这个结论不一定是对的

下面我们通过一列的论证

  • 首先我们知道Java执行指令要么通过解释器或者编译器后的本地代码

  • 那源代码到最终的指令集需要经过：

    源代码->编译器优化重排序->指令级并行重排序->内存系统重排序->最终执行顺序的指令顺序

  • 所以猜测是不是因为编译器指令的原因？我们去掉volatile加上VM参数 -Xint（ps：只启用解析器执行代码）,结果发现程序顺利停止了

回顾下Volatile关键字知识点

  • Java中的volatile关键字的作用：保证资源的可见性、保证基本数据类型资源的原子性
  • 虚拟机中基于原语指令lock实现原子性，基于内存屏障保障可见性，同时禁止编译器指令重排序(禁止了编译器优化)
  • 编译器优化涉及指令重排序、是否从寄存器中获取资源

结论

  • 之所以添加Volatile能停止，不是因为禁止指令重排序，而是禁止了编译器优化.
  • 只启用解释器执行能停止，计算机中每个CPU都有自己的缓存，基于JMM，每个线程都有自己的本地内存，这儿主线程修改了本地内存资源，主线程中本地内存资源总会在某个时刻更新到主存中，从而CPU通过嗅探总线会通知CPU缓存资源失效，从而重新到主存中获取

扩展

  • C语言中也有volatile关键字，Linux的开发者极其不推荐使用，因为他们认为大多数使用场景都违背了此关键字设计的初衷
